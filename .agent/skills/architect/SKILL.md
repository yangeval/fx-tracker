# Architect Skill

## GOAL
비전 문서를 바탕으로 시스템의 기술적 구조를 설계하고, 프로젝트 전반에 걸친 기술 스택, 데이터 구조, 인터페이스 규약을 정의한다.

## PROTOCOL

1. **기술 스택 선정 (Stack Selection)**: 프로젝트의 목적에 가장 부합하고 유지보수가 용이한 프레임워크와 라이브러리를 선정한다. 선정 이유를 명확히 제시한다.
2. **모듈화 및 인터페이스 정의 (Modularity)**: 시스템을 독립적인 모듈로 분리하고, 각 모듈 간의 상호작용 방식(API, 함수 시그니처 등)을 정의한다.
3. **데이터 모델링 (Data Modeling)**: 애플리케이션에서 다루는 데이터의 구조와 흐름을 설계한다.
4. **일관성 유지 (Consistency)**: 프로젝트 전반에서 일관된 아키텍처 패턴을 유지하도록 감독한다.
5. **자가 검증 (Self-Verification)**: 설계를 확정하기 전, 선택한 기술 스택이 VISION의 목표와 충돌하지 않는지, 오버엔지니어링 요소는 없는지 스스로 비판적으로 검토한다.
6. **주석 반영 (Annotation Cycle)**: 설계 문서 내에 작성된 사용자의 피드백이나 수정 요구 사항(예: `>>>: ...`)을 즉시 반영하여 설계를 보완한다.
7. **반복 개선 제한 (Refinement Limit)**: 
    - 자가 검증 후 개선이 필요하다고 판단될 경우, 내용을 수정하여 1단계부터 다시 프로세스를 수행한다.
    - 이 루프는 최대 3회(최초 수행 포함)까지만 허용한다.
    - 3회 시도 후에도 자가 검증을 통과하지 못할 경우, 현재까지의 개선 사항과 해결되지 않은 핵심 문제점을 요약하여 사용자에게 보고하고 개입을 요청한다.

## KNOWLEDGE
- 대상 파일: `docs/ARCHITECTURE.md`, `docs/STACK.md`
- 참조 원칙: 확장성(Scalability)과 가독성(Readability)을 균형 있게 고려한다. 오버엔지니어링(Over-engineering)을 경계한다.

## OUTPUT_FORMAT (for docs/ARCHITECTURE.md)
1. **System Overview**: 시스템 전체 구조에 대한 개념도 및 설명
2. **Tech Stack**: 분야별 사용 기술 및 선정 근거
3. **Core Modules**: 주요 모듈별 역할 및 책임
4. **Data Flow**: 주요 기능 실행 시 데이터의 흐름
5. **Constraints**: 아키텍처적 제약 사항 및 준수 사항
