# Vision Skill

## GOAL
사용자의 머릿속에 있는 단편적이고 모호한 아이디어를 대화를 통해 추출하고, 이를 구체적이고 실행 가능한 **프로젝트 비전(Vision)**으로 정립한다.

## PROTOCOL

1. **질문 기반 탐색 (Inquiry-First)**: 사용자의 설명을 듣고 즉시 결론을 내리지 않는다. 항상 "왜(Why)", "누가(Who)", "어떻게(How)"에 대한 질문을 최소 1~2개 던져 모호한 부분을 제거한다.
2. **반복적 구체화 (Iterative Refinement)**: 사용자의 답변을 기반으로 내용을 정리하고, 에이전트가 알고 있는 기술적 지식이나 베스트 프래티스를 보완하여 다시 제안한다.
3. **영속적 기록 (Persistence)**: 대화의 최종 합의 결과는 반드시 `docs/VISION.md` 파일에 기록하여 프로젝트의 북극성(North Star)이 되게 한다.
4. **대화 종결 조건**: 사용자가 정리된 비전에 대해 "검토 완료" 또는 "확정"의 의사를 표시할 때까지 반복한다.
5. **주석 반영 (Annotation Cycle)**: 사용자가 문서 내에 인라인 주석(예: `>>>: ...`)이나 특정 섹션을 통해 피드백을 남긴 경우, 이를 최우선적으로 반영하여 내용을 갱신한다.
6. **자가 검증 (Self-Verification)**: 질문을 던지기 전, 해당 질문이 비전의 구체화에 반드시 필요한지 스스로 검토한다. 사용자의 답변이 모순될 경우 이를 지적하고 다시 확인한다.
7. **반복 개선 제한 (Refinement Limit)**: 
    - 자가 검증 후 개선이 필요하다고 판단될 경우, 내용을 수정하여 1단계부터 다시 프로세스를 수행한다.
    - 이 루프는 최대 3회(최초 수행 포함)까지만 허용한다.
    - 3회 시도 후에도 자가 검증을 통과하지 못할 경우, 현재까지의 개선 사항과 해결되지 않은 핵심 문제점을 요약하여 사용자에게 보고하고 개입을 요청한다.

## KNOWLEDGE
- 대상 파일: `docs/VISION.md`
- 참조 원칙: 사용자의 비즈니스 가치와 해결하고자 하는 문제의 본질에 집중한다. 현실적인 기술 제약은 ARCHITECT 단계에서 다루되, 여기서는 '결과물의 모습'에 집중한다.

## OUTPUT_FORMAT (for docs/VISION.md)
1. **Project Name**: 프로젝트의 가칭 또는 확정 명칭
2. **Core Value**: 이 프로젝트가 존재하는 이유 (단 한 문장)
3. **Problem Statement**: 해결하고자 하는 기존의 불편함이나 문제점
4. **Main Features**: 핵심 기능 리스트
5. **Expected Outcome**: 프로젝트 완료 후 변화될 미래의 모습
