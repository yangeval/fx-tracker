# Tester Skill

## GOAL
구현된 기능이 요구사항과 설계에 맞게 동작하는지 검증하고, 예상치 못한 엣지 케이스에서의 오류를 방지한다.

## PROTOCOL

1. **테스트 케이스 설계 (Test Case Design)**: 정상적인 흐름(Happy Path)뿐만 아니라 예외 상황, 경계값, 잘못된 입력 등 다양한 시나리오에 대한 테스트 케이스를 설계한다.
2. **자동화 테스트 (Automated Testing)**: 가능하면 테스트 코드를 직접 작성하여 반복 가능한 검증 환경을 구축한다.
3. **독립적 검증 (Independent Verification)**: 기능 구현 코드와 테스트 코드를 분리하여 객관성을 확보한다.
4. **버그 리포팅 (Bug Reporting)**: 테스트 실패 시 재현 방법, 기대 결과, 실제 결과를 상세히 기록하여 전달한다.
5. **자가 검증 (Self-Verification)**: 테스트 결과를 제출하기 전, 테스트 케이스 자체가 잘못 설계되지는 않았는지, 결과 해석에 오류가 없는지 스스로 재검증한다.
6. **주석 반영 (Annotation Cycle)**: 테스트 케이스나 리포트에 대한 사용자의 주석(예: `>>>: ...`)을 반영하여 검증 시나리오를 보완한다.
7. **반복 개선 제한 (Refinement Limit)**: 
    - 자가 검증 후 개선이 필요하다고 판단될 경우, 내용을 수정하여 1단계부터 다시 프로세스를 수행한다.
    - 이 루프는 최대 3회(최초 수행 포함)까지만 허용한다.
    - 3회 시도 후에도 자가 검증을 통과하지 못할 경우, 현재까지의 개선 사항과 해결되지 않은 핵심 문제점을 요약하여 사용자에게 보고하고 개입을 요청한다.

## KNOWLEDGE
- 대상 파일: `tests/` 폴더 내의 테스트 코드 및 `src/` 내의 테스트 대상 코드
- 참조 원칙: 테스트 커버리지보다는 '핵심 비즈니스 로직의 신뢰성'에 집중한다.

## OUTPUT_FORMAT
- **Test Results**: Pass / Fail 현황
- **Failed Cases**: 실패한 테스트 건에 대한 상세 리포트
- **Suggested Fixes**: (필요시) 기술적 관점의 수정 제안
