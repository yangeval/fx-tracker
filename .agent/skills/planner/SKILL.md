# Planner Skill

## GOAL
승인된 시스템 설계안을 바탕으로 실제 구현이 가능한 가장 작은 단위의 작업 목록(Task List)을 작성하고 진행 상태를 관리한다.

## PROTOCOL

1. **작업 원자화 (Atomicity)**: 각 작업은 한 번의 코드 수정 및 제출로 완료될 수 있을 만큼 충분히 작아야 한다. 작업이 크다면 더 하위 단계로 쪼갠다.
2. **의존성 명시 (Dependency Tracking)**: 각 작업이 선행되어야 하는 조건과 성공 기준(Definition of Done)을 명확히 정의한다.
3. **상태 업데이트 (Status Management)**: 모든 구현 단계에서 `TASK_BOARD.md` 또는 계획 문서 내의 Todo 리스트를 실시간으로 업데이트하여 현재 진행 상황을 투명하게 유지한다.
4. **리스크 식별 (Risk Identification)**: 구현 과정에서 발생할 수 있는 기술적 난관이나 병목 지점을 사전에 파악하여 관리자에게 보고한다.
5. **자가 검증 (Self-Verification)**: 작업 목록을 제출하기 전, 각 태스크가 실제로 '원자적'인지, 누락된 선행 작업은 없는지 스스로 시뮬레이션해 본다.
6. **주석 반영 (Annotation Cycle)**: 계획서 내의 태스크나 접근 방식에 대해 사용자가 남긴 주석(예: `>>>: ...`)을 바탕으로 작업 우선순위나 범위를 조정한다.
7. **반복 개선 제한 (Refinement Limit)**: 
    - 자가 검증 후 개선이 필요하다고 판단될 경우, 내용을 수정하여 1단계부터 다시 프로세스를 수행한다.
    - 이 루프는 최대 3회(최초 수행 포함)까지만 허용한다.
    - 3회 시도 후에도 자가 검증을 통과하지 못할 경우, 현재까지의 개선 사항과 해결되지 않은 핵심 문제점을 요약하여 사용자에게 보고하고 개입을 요청한다.

## KNOWLEDGE
- 대상 파일: `docs/` 폴더 내의 각 프로젝트 계획서 및 `TASK_BOARD.md`
- 참조 원칙: 구현의 복잡도를 낮추기 위해 '단순함(Simplicity)'을 최우선 가치로 둔다.

## OUTPUT_FORMAT (for plan.md)
1. **Context**: 현재 구현하려는 기능의 배경 및 목적
2. **Implementation Plan**: 단계별 접근 방법 (Phase 1, 2, 3...)
3. **Todo List**: 
    - [ ] Task 1: 상세 내용 (기대 결과 포함)
    - [ ] Task 2: 상세 내용 (기대 결과 포함)
4. **Verification Strategy**: 구현 후 검증 방법
